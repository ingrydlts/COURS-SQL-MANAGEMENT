Module 01

Modèle Conceptuel
des Données

Le MCD (ou modèle Entité/Association) permet de décrire la réalité AVANT de toucher à une seule ligne de SQL. C'est la phase de réflexion.

📦 Qu'est-ce qu'une Entité ?

Une entité type = une catégorie d'objets du monde réel qui partagent les mêmes caractéristiques.
Une occurrence = une instance concrète de cette entité.

Entité Type

Représentation abstraite : Étudiant, Voiture, Cours. C'est le "moule".

Occurrence (instance)

Un étudiant précis : "Marie Curie, n°345". C'est une ligne dans la future table.

Représentation

Rectangle avec le nom en haut, les attributs en dessous. L'identifiant est souligné.

Exemple — Entité ÉTUDIANT

ÉTUDIANT
🔑 numEtudiant · nomEtudiant · prenomEtudiant · dateNaissance

🔑 = identifiant unique (souligné dans les diagrammes)

🏷️ Les 4 types d'attributs

Attribut simple

Valeur unique et directe. Ex : nom, prénom, prix.

Attribut multivalué ⚠️

Peut contenir plusieurs valeurs (ex: plusieurs auteurs pour un livre). → Il faut en faire une entité séparée !

Attribut composé

Formé de sous-attributs : adresse = rue + CP + ville. Souvent transformé en entité.

Attribut dérivé ⚠️

Calculé depuis d'autres attributs. Ex : âge calculé depuis dateNaissance. → Ne pas stocker dans la BDD !

⚠️
Les attributs multivalués et dérivés ne peuvent pas exister dans une base de données finale. Il faut les transformer.
🔗 Les Associations

Une association représente un lien entre entités, défini par un verbe à l'infinitif.
Elle peut elle-même porter des attributs si ces données décrivent le lien.

Étudiant — suit → Cours (avec attributs de relation)

🔢 Les Cardinalités

La cardinalité d'une association = le nombre de fois minimal et maximal qu'une entité participe à l'association.
Format : (min, max)

CardinalitéSignificationExemple
(0,1)Optionnel, au plus 1Un employé peut avoir 0 ou 1 voiture de fonction
(1,1)Obligatoire, exactement 1Un étudiant est inscrit dans exactement 1 université
(0,N)Optionnel, plusieurs possibleUn cours peut avoir 0 ou N étudiants inscrits
(1,N)Obligatoire, plusieurs possibleUne université a au moins 1 étudiant
💡
Astuce lecture : La cardinalité côté A décrit la participation de A dans l'association vue depuis B. "Un CLIENT passe (1,N) commandes" = un client peut passer 1 à N commandes.

📐 Les Formes Normales

Les formes normales garantissent la qualité et la cohérence du modèle. Chaque niveau inclut le précédent.

BCNF — Forme Boyce-Codd : seules les DF élémentaires depuis un identifiant
3FN — Toutes les dépendances fonctionnelles sont directes
2FN — Toutes les DF sont élémentaires (dépend de tout l'identifiant)
1FN — Chaque attribut est atomique (1 seule valeur) + identifiant présent
⚠️
3FN en pratique : toute propriété doit dépendre uniquement de l'identifiant de son entité, pas d'un autre attribut. Ex : dans VOITURE(Immatriculation, Type, Marque), la Marque dépend du Type → violation 3FN → créer l'entité MODÈLE séparée.

L'entreprise ACME veut un catalogue numérique. Chaque produit a : nom, référence, prix, photo. Les produits appartiennent à des catégories (nom uniquement). Les clients professionnels peuvent commander des produits. Le règlement se fait sur facture.

→ Identifie les entités, leurs attributs et leurs identifiants.

▶ Voir la correction
5 entités :

🔹 PRODUIT : Reference (PK), Nom, Prix, Photo, IdCategorie (FK)
🔹 CATEGORIE : IdCategorie (PK), NomCategorie
🔹 CLIENT : IdClient (PK), NomClient
🔹 COMMANDE : NumCommande (PK)
🔹 FACTURE : NumFacture (PK)

Associations :
· Produit — appartient — Catégorie (1-N)
· Client — passer — Commande (1-N)
· Commande — contenir — Produit (N-N) → table CONTIENT
· Commande — donner lieu — Facture (1-1)
✏️ Exercice 2 — Cardinalités

Pour chacune des situations suivantes, donne la cardinalité correcte :

a) Un appartement appartient à un et un seul immeuble. Un immeuble peut avoir 0 ou N appartements.
b) Un étudiant s'inscrit dans une et une seule université. Une université accueille 1 à N étudiants.
c) Un employé peut avoir 0 ou N voitures. Une voiture doit obligatoirement avoir 1 propriétaire.

▶ Voir la correction
a) Appartement (1,1) — Fait partie — (0,N) Immeuble

b) Étudiant (1,N) — Est inscrit — (1,1) Université
(L'université accueille 1..N étudiants donc côté université : 1,N ; l'étudiant est dans 1 seule université donc son côté : 1,1)

c) Voiture (1,N) — Possède — (0,N) Employé
(La voiture a obligatoirement 1 propriétaire → min=1 côté voiture)
✏️ Exercice 3 — Formes normales

La table VOITURE(Immatriculation, Couleur, Type, Marque, Puissance) est-elle en 3FN ? Pourquoi ? Si non, corrige-la.

▶ Voir la correction
Non, elle n'est pas en 3FN.

Problème : Type → Marque, Puissance — or Type n'est pas l'identifiant (Immatriculation l'est).
Marque et Puissance dépendent de Type (attribut non-identifiant) → violation 3FN.

Correction :
→ VOITURE(Immatriculation, Couleur, Type_FK)
→ MODELE(Type, Marque, Puissance)
Association : Voiture — De type — Modèle (1,1)-(0,N)
Module 02

MCD →
Modèle Logique

Le MLD transforme le MCD en tables concrètes. Il existe 3 règles de transformation selon la cardinalité de chaque association.

📋 Les 3 Règles de Transformation
Cas MCDTransformation MLDContrainte
1 — N La PK du côté "1" migre en FK côté "N" Si min=1 côté N → FK NOT NULL
N — N L'association devient une table à part entière avec les 2 PK comme clé composite Attributs de l'association dans cette table
1 — 1 La FK migre vers le côté "dépendant" avec UNIQUE FK UNIQUE NOT NULL

3️⃣ Règle 3 — Association 1-1
💡
La FK migre vers le côté dépendant avec la contrainte UNIQUE pour garantir le 1-1. Ex : une Commande génère exactement 1 Facture → la FK va dans FACTURE avec UNIQUE.
MLD
-- Commande génère 1 Facture (1-1) COMMANDE(NumCommande PK, IdClient FK, ...) FACTURE(NumFacture PK, NumCommande FK UNIQUE NOT NULL, ...)
🔄 Associations Réflexives

Une entité peut avoir une relation avec elle-même. Ex : un Employé a un supérieur qui est aussi un Employé.

✅
En MLD, on garde une seule table mais on ajoute un attribut auto-référençant :
EMPLOYE(idEmploye, nom, idSupervieur FK → EMPLOYE)

✏️ Exercice 1 — Transformer MCD en MLD

Donne le MLD complet pour : CLIENT (0,N) — passer — (1,1) COMMANDE (1,N) — contenir — (0,N) PRODUIT. La relation contenir a l'attribut "quantite".

▶ Voir la correction
CLIENT(idClient, nomClient)
COMMANDE(numCommande, idClient FK NOT NULL)
→ Règle 1-N : PK de CLIENT migre dans COMMANDE (côté N)

PRODUIT(reference, nom, prix)

CONTIENT(numCommande FK, reference FK, quantite)
→ Règle N-N : l'association devient une table, PK composite
✏️ Exercice 2 — Cas pratique

Un PATIENT (SECU PK) cotise à une MUTUELLE (codeMutuelle PK). Cardinalités : Patient (0,1) — Mutuelle (1,N). Donne le MLD.

▶ Voir la correction
Association 1-N (une mutuelle a plusieurs patients, un patient a au plus 1 mutuelle).

MUTUELLE(codeMutuelle, nomMutuelle, adresseMutuelle)
PATIENT(SECU, nomPatient, prenomPatient, adressePatient, codeMutuelle FK)

La FK est nullable (pas NOT NULL) car la cardinalité min côté Patient est 0 → un patient peut ne pas avoir de mutuelle.
✏️ Exercice 3 — Identifier l'erreur

Quelqu'un propose : COMMANDE(numCommande) et FACTURE(numFacture). La relation est 1-1. Il ne met pas de FK. Qu'est-ce qui manque et où doit-elle aller ?

▶ Voir la correction
Il manque la FK pour matérialiser la relation 1-1.

Solution : mettre la FK dans FACTURE (côté dépendant) :
FACTURE(numFacture, numCommande FK UNIQUE NOT NULL, montant, date)

Le UNIQUE garantit qu'une commande ne peut être liée qu'à une seule facture.
Module 03

MPD & MySQL

Le Modèle Physique des Données traduit le MLD dans le langage d'un SGBD précis : ici MySQL. On définit les types, les contraintes, et on écrit le SQL.

📏 Conventions MySQL

Nommage

Tout en minuscules, alphabet anglais, underscores à la place des espaces. Max 20 caractères (convention Oracle).

Casse

MySQL est sensible à la casse : student ≠ Student. Toujours utiliser les minuscules.

Préfixage

Préfixer les attributs par un abrégé du nom de table : std_name (student), unv_name (university).

💡
Le préfixage permet d'identifier immédiatement les clés étrangères dans les requêtes multi-tables. unv_name dans student → c'est une FK vers university.
🗃️ Les Types de Données MySQL
VARCHAR(n)
Chaîne de caractères de longueur variable. n = nombre max de caractères.
std_name VARCHAR(30)
ENUM('a','b',...)
Liste prédéfinie de valeurs autorisées.
size ENUM('S','M','L')
INT / INTEGER
Entier standard. Plage : -2 147 483 648 à 2 147 483 647.
std_id INT
TINYINT
Petit entier : -128 à 127. Utile pour les âges, notes…
note TINYINT
DECIMAL(p,s)
Nombre exact. p = précision totale, s = chiffres après virgule.
prix DECIMAL(10,2)
DATE / DATETIME
Date seule ou date+heure.
date_debut DATE
⌨️ Les Commandes SQL Essentielles

Voici la structure complète pour créer et peupler une base. L'ordre est crucial !

CREATE DATABASE
CREATE DATABASE education;
CREATE TABLE
CREATE TABLE university( unv_name VARCHAR(50) NOT NULL PRIMARY KEY, unv_country VARCHAR(20) NOT NULL, unv_address VARCHAR(90) NOT NULL ); CREATE TABLE student( std_id INT NOT NULL PRIMARY KEY, unv_name VARCHAR(50), std_name VARCHAR(30) NOT NULL, std_nationality VARCHAR(20) NOT NULL, FOREIGN KEY (unv_name) REFERENCES university(unv_name) );
⚠️
Ordre de création ! Toujours créer la table "parent" (university) AVANT la table "enfant" (student) qui la référence. Sinon : erreur de FK.
INSERT INTO
INSERT INTO university VALUES ('UTT', 'France', 'Troyes'), ('PSB', 'France', 'Paris'); INSERT INTO student VALUES (123, 'UTT', 'Alexander', 'American'), (345, 'PSB', 'Marie', 'French');
UPDATE
UPDATE university SET unv_address = 'San Francisco' WHERE unv_name = 'Stanford';
ALTER TABLE
ALTER TABLE student ADD std_age INTEGER;

✏️ Exercice 1 — Quel type choisir ?

Pour chacun, choisis le type MySQL le plus adapté :
a) Le prix d'un produit (ex: 12.99€)
b) Le nom d'un client (max 50 caractères)
c) La taille d'un vêtement parmi XS/S/M/L/XL
d) L'identifiant auto-incrémenté d'un étudiant
e) La date de naissance

▶ Voir la correction
a) DECIMAL(10,2) — pour les montants financiers exacts
b) VARCHAR(50) — chaîne de longueur variable
c) ENUM('XS','S','M','L','XL') — liste prédéfinie

✏️ Exercice 2 — Écrire le CREATE TABLE

Écris le SQL pour créer la table PRODUIT avec : reference (PK, VARCHAR 20), nom (VARCHAR 50, obligatoire), prix (décimal 10,2), idCategorie (FK vers CATEGORIE).

▶ Voir la correction 
CREATE TABLE produit(
  pro_reference VARCHAR(20) NOT NULL PRIMARY KEY,
  pro_nom       VARCHAR(50) NOT NULL,
  pro_prix      DECIMAL(10,2),
  cat_id        INT,
  FOREIGN KEY (cat_id) REFERENCES categorie(cat_id)
);
Note : il faut avoir créé la table categorie AVANT produit !
✏️ Exercice 3 — Debug SQL

Ce code génère une erreur. Pourquoi ?
DROP TABLE university; alors que student a une FK vers university.

▶ Voir la correction
Erreur : contrainte d'intégrité référentielle violée.

La table student a une FOREIGN KEY qui référence university. MySQL interdit de supprimer une table si d'autres tables la référencent.

Solution : supprimer d'abord la table enfant :
DROP TABLE student;
DROP TABLE university;
Module 04

Requêtes SQL

On travaille sur une base de location de films (~1000 films) avec les tables : film, customer, payment, inventory, actor, staff. Toutes les requêtes s'écrivent sur papier à l'examen.

1️⃣ SELECT — La base de tout
SELECT colonne1, colonne2 ou * pour tout afficher
FROM nom_table table source
WHERE condition optionnel
ORDER BY colonne ASC/DESC optionnel
LIMIT n optionnel
Exemples pratiques
-- Tous les titres de films SELECT title FROM film; -- ID, nom, prénom, email des clients SELECT customer_id, last_name, first_name, email FROM customer; -- Films par ordre alphabétique inverse SELECT title FROM film ORDER BY title DESC; -- Les 10 premiers films SELECT * FROM film LIMIT 10;
2️⃣ WHERE — Filtrer les résultats
OpérateurSignificationExemple

Exemples
-- Films dont le titre contient "LOVE" SELECT title FROM film WHERE title LIKE '%LOVE%'; -- Films de durée entre 90 et 120 min SELECT title, length FROM film WHERE length BETWEEN 90 AND 120; -- Clients avec email SELECT first_name, last_name FROM customer WHERE email IS NOT NULL;
3️⃣ Fonctions d'Agrégation

COUNT(*)

Compte le nombre de lignes.

SELECT COUNT(*) FROM film;

SUM(col)

Somme des valeurs d'une colonne.

SELECT SUM(amount) FROM payment;

AVG(col)

Moyenne des valeurs.

SELECT AVG(rental_rate) FROM film;

MAX / MIN

Valeur maximale ou minimale.

SELECT MAX(length) FROM film;
4️⃣ DISTINCT & GROUP BY
DISTINCT + GROUP BY
-- Types de notations uniques dans la BDD SELECT DISTINCT rating FROM film; -- Nombre de films par notation (rating) SELECT rating, COUNT(*) AS nb_films FROM film GROUP BY rating ORDER BY nb_films DESC; -- Total des paiements par client SELECT customer_id, SUM(amount) AS total FROM payment GROUP BY customer_id ORDER BY total DESC;
💡
Règle : tout ce qui est dans le SELECT et qui n'est pas une fonction d'agrégation doit être dans le GROUP BY.

✏️ Exercice 4 — LIKE & BETWEEN

b) Trouve les films dont le rental_rate est entre 1.00 et 3.00.

▶ Voir la correction

FROM actor

-- b)
SELECT title, rental_rate
FROM film
WHERE rental_rate BETWEEN 1.00 AND 3.00;
✏️ Exercice 5 — Examen type (toutes compétences)

Écrire en SQL : "Donne le nombre de films par notation (rating), uniquement pour les notations ayant plus de 200 films, triés par nombre décroissant."

▶ Voir la correction 
SELECT rating, COUNT(*) AS nb_films
FROM film
GROUP BY rating
HAVING COUNT(*) > 200
ORDER BY nb_films DESC;
Nota : HAVING filtre les groupes (après GROUP BY), contrairement à WHERE qui filtre les lignes. C'est un point avancé — si tu veux aller plus loin !
